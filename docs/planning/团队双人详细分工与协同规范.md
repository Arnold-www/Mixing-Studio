# 调音室团队双人详细分工与协同规范

**基于严格 MVVM 架构的双人操控 AI 协同开发规范**

为了在 7 月 18 日截止前完成一个功能完整、结构严谨、可稳定运行、且能明确体现 MVVM 的调音项目，本项目采用“两名成员分别操控 AI、各自主责对应层、阶段性交叉测试、通过后合入主分支”的协同方式。AI 负责高频生成和补全，人类负责分层边界、代码验收、互相测试、版本控制和报告证据留存。

## 一、 角色定义与核心边界

### 成员 A：操控 AI 负责 Model / DSP / Persistence
*   **能力侧重：** C++ Model、DSP 算法、音频导入/混音/导出、SQLite/JSON 持久化、底层测试。
*   **主责边界：** 负责 `src/Model/`、`src/DSP/` 和持久化相关模块，不直接改 QML 业务逻辑；所有对外能力必须通过 ViewModel 契约暴露。
*   **报告价值：** 说明多轨混音、Pan、EQ、Compressor、Limiter、波形/VU/频谱数据、工程保存/导出等底层实现。

### 成员 B：操控 AI 负责 ViewModel / View / Report
*   **能力侧重：** C++ ViewModel、`Q_PROPERTY`、状态机、QML UI、可视化控件、报告结构和证据整理。
*   **主责边界：** 负责 `src/ViewModel/`、`src/View/` 和报告材料，不直接操作 PCM 字节流；所有 UI 操作必须通过 ViewModel 调用 Model。
*   **报告价值：** 说明 MVVM 绑定、轨道状态机、Solo/Mute 优先级、UI 数据绑定、波形/VU/频谱展示和报告证据链。

### 交叉测试责任：互测对方实现
*   **成员 A 测试 B：** 检查 ViewModel/QML 是否越过 Model、属性绑定是否正确、UI 状态是否能正确触发底层接口。
*   **成员 B 测试 A：** 检查 Model/DSP 是否按接口返回数据、音频参数是否生效、保存/导出/分析数据是否能被 ViewModel 消费。
*   **合入原则：** 每个阶段必须由非主责成员测试通过并留下记录后，才能合入主分支。

## 三、 截至 7 月 18 日的 AI 主导高完成度冲刺流水线

在截止日前，两名成员分别操控 AI 推进各自主责层：成员 A 聚焦 Model/DSP/Persistence，成员 B 聚焦 ViewModel/View/Report。每天必须形成一个可运行版本，并进行至少一次交叉测试。**关键节点：第 5 天完成完整调音闭环，第 8 天完成素材/工程闭环，7 月 17 日冻结代码与报告内容，7 月 18 日最终提交。**

### 🔄 阶段一：基建与发声 (Day 1)

*   **成员 A + AI：** 搭建 CMake、`AudioEngine`、`DspProcessor`、基础 Model 接口和测试骨架。
*   **成员 B + AI：** 搭建 QML 主界面、`TrackViewModel`、`MixerViewModel`、`ProjectViewModel`、Mock 数据和报告截图模板。
*   **交叉测试目标：** A 测试 B 的 ViewModel/QML 是否只通过接口调用；B 测试 A 的 Model/DSP 接口是否能被 Mock 或 VM 正确调用。通过后合入主分支。

### 🔄 阶段二：播放闭环 (Day 2~3)

*   **成员 A + AI：** 跑通本地音频导入、多轨加载、播放状态、Seek、Loop、主音量底层接口。
*   **成员 B + AI：** 完成轨道列表、播放控制区、主音量控件、进度条和状态绑定。
*   **交叉测试目标：** B 使用 UI 测试 A 的播放接口；A 检查 B 的播放命令是否正确传入 Model。通过后合入主分支。

### 🔄 阶段三：状态与防线 (Day 4~5) 🚩 中期验收

*   **成员 A + AI：** 实现多轨线性混音、主输出限幅、轨道音量、Pan、三段 EQ、Compressor、Bypass 的底层处理。
*   **成员 B + AI：** 实现 Mute、Solo、Pan、EQ、Compressor、Bypass 的 ViewModel 状态机和 QML 控件。
*   **交叉测试目标：** B 试听 A 的底层处理是否生效；A 检查 B 的状态机和参数范围是否正确。通过后合入主分支。

### 🔄 阶段四：视觉与轻量算法 (Day 6~8)
*   **成员 A + AI：** 手写波形降采样、VU 电平计算、简易频谱数据、峰值/削波提示；实现 SQLite 素材库和 JSON 工程保存/加载。
*   **成员 B + AI：** 实现波形、进度线、VU 表、频谱图、素材搜索 UI、最近打开记录和工程恢复入口。
*   **交叉测试目标：** B 验证 A 的分析数据和工程数据能被 UI 消费；A 检查 B 的可视化数据来源和工程状态回填是否正确。通过后合入主分支。

### 🔄 阶段五：工程与最终交付 (Day 9~10 / 7月17日~7月18日) 🚩 最终提交
*   **成员 A + AI：** 完成 WAV 导出验收、DSP 单元测试、底层架构检查清单和底层部分分报告。
*   **成员 B + AI：** 完成 Mock 验证模式、样例工程、报告截图清单、总报告结构和 UI/VM 部分分报告。
*   **交叉测试目标：** A 测试最终 UI/VM 流程，B 测试最终 Model/DSP/导出结果。双方签字确认后合入主分支并封版。

## 四、 分支、互测与合入铁律 (Workflow)

1.  **接口先行协议 (Contract First)：**
    在每个阶段开始前，两名成员先共同确认 ViewModel 的属性、信号、槽函数和数据格式，再分别操控 AI 实现各自主责层。接口稳定可以减少返工，但每天仍要运行验收。后端算法未完成时必须使用 Mock 数据保证 UI、测试和报告截图不被阻塞。
2.  **物理目录隔离：**
    `src/Model/` 与 `src/DSP/` 只处理数据、音频和持久化；`src/ViewModel/` 只处理状态、命令和属性通知；`src/View/` 只处理 QML 渲染与事件绑定。
3.  **阶段分支开发：**
    *   成员 A 使用 `feature/A-model-dsp-*` 或等价分支开发 Model/DSP/Persistence。
    *   成员 B 使用 `feature/B-vm-view-*` 或等价分支开发 ViewModel/View/Report。
    *   禁止未测试代码直接进入 `main` 分支。
4.  **交叉测试与人工验收：**
    *   成员 A 每阶段必须测试成员 B 的 ViewModel/QML 是否符合接口、是否存在越层调用、是否能正确触发底层。
    *   成员 B 每阶段必须测试成员 A 的 Model/DSP 是否按接口工作、数据是否可消费、导出和保存是否正确。
    *   双方互测通过后，记录测试结果、问题修复和对应提交，再合入主分支。
    *   7 月 17 日进入冻结期后停止新增高风险功能，只修复稳定性、测试、报告材料和大模型使用记录。
5.  **大模型使用记录：**
    *   每次使用 AI 完成需求分析、代码生成、调试、测试或文档整理时，都要记录使用阶段、采用模式、提示词摘要、输出内容和人工验收结果。
    *   总报告中汇总大模型使用方式，个人分报告中说明个人如何使用 AI、如何验证 AI 输出、哪些内容由人工修改。
6.  **版本控制与报告证据：**
    *   成员应保持持续有效提交，提交信息对应具体功能、测试、修复或文档。
    *   每次合入主分支前，必须能对应到功能实现提交、对方测试提交或测试记录、问题修复提交。
    *   报告中保留提交截图或提交日志摘要，用于证明成员协作紧密、过程可追溯。
    *   报告中必须包含单元测试、其他测试、持续集成或构建工具的使用说明。
7.  **总报告与分报告要求：**
    *   总报告必须覆盖项目背景、需求分析、总体架构、MVVM 设计、核心功能、测试结果、工具链、大模型使用记录和项目总结。
    *   成员分报告必须说明个人负责内容、有效提交、AI 使用方式、人工修改内容、测试证据和遇到的问题。
    *   大模型使用记录必须保留“采用模式”，例如：AI 主导代码生成、AI 辅助调试、AI 生成测试、AI 审查架构、人工最终验收。
    *   报告中的功能截图、测试截图、提交截图和 AI 使用记录应能相互对应，避免报告内容与实际工程脱节。
